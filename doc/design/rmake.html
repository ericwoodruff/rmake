What is rmake?

rmake (rsync make) is a distributed make tool that synchronizes a master SVN tree view to any build resource that you have SSH access to and remotely invokes make on that build resource. It is designed to be run on your workstation, rather than on a specific build server.

Benefits

  • Copy the master workspace to one or more build locations.
  • Clean one or more build locations.
  • Build in one or more build locations.
  • Works like make from the current working directory on one's workstation.
  • Can build multiple platforms serially or in parallel.
  • Bash auto-completion support.
  • Extensible to any number of platforms.
  • Encapsulates build details like:
    • Setting up KCACHE
    • Initializing env on Windows
  • Separates the build trees from the developer's sandbox; keeps the svn tree pure.
  • Supports emailing results from parallel builds.
  • Understands svn status to exclude files that are unversioned or in poor status (conflicted).
  • By default does not transfer svn metadata (which includes the base revision copy).
  • Can validate build servers meet build requirements:
    • Proper RPMs installed
    • Password-less ssh access
    • VM has latest version

User feedback

"I use rmake all the time to make sure I don't break builds on any of the platforms – saved me a lot of time and typing. And it's so nice to maintain only one SVN snapshot, and have rmake replicate it when necessary." - Oleg Katsitadze, Beaverton, Oregon

"Since I am working on Big-Bend builds, I use rmake to be able to check that my changes work on both the big-bend build server and the guiness build server. I can do it with one simple command using rmake! This saves me a lot of time." - Yashjeet Kaur, Beaverton, Oregon

Architecture

How it works

rmake uses a master/shadow architecture and intentionally does not transfer any SVN meta-data to build resources. Therefore, changes to a specific platform cannot be committed from a build resource, but rather from the master working copy instead. The idea is a sort of 'write once, build everywhere' philosophy.
This approach has the great advantage that the master copy never gets polluted with build derived files, making it extremely easy to see what source files are new and what has been changed using SVN; this means that the svn:ignore property has virtually no use to rmake users.

Note: rmake copies files to the build servers with read-only permissions to discourage and prevent accidental changes of files on the server that are not suited to be committed back to the svn repository.

rmake and Disposable Build VMs

Build VMs are disposable commodities. So why shouldn't any data on them be treated the same way? rmake does this by separating the build trees from the source tree with efficient replication via rsync. You lost a disk in your host machine? Switch to a new cluster host, set up another VM, startup your build server and run rmake again. In an almost self-healing way, you've got your build running again without loss of any work because it wasn't really there in the first place, only a reflection of your work was on that disk. Your real SVN workspace is safe and sound on a secure resource like the ds cluster.

Hypothetical Checkouts

rmake can clean the build trees to match the master copy exactly. It can even go further and refuse to copy, or even delete from the shadow copy, the files that are not in "good standing" with SVN (such as unversioned, ignored or conflicted files). This allows a build to be run on a build server as if it was checked out from the changes pending in the master workspace/sandbox. This helps protect against the 'I forgot to checkin some files' case.

Is/Is Not

rmake can be used:

  • to make a single module on a single platform
  • to make a single module on multiple platforms, in serial or parallel
  • to build the entire klondike tree (minus kernel stuff at the moment) including external on multiple platforms in serial or parallel.
  • to validate changes as a pre-commit step
  • to efficiently produce bits for multiple architectures to test in a mixed 32-bit/64-bit environment
  • to validate that changes build from scratch (by cleaning all build-derived resources beforehand)
  • to sanity check compilation of all platforms from a single svn working copy

rmake is not:

  • project, architecture, build server or VM specific
  • just for building remotely, separation between your svn tree and build tree is always a good idea
  • any different than running 'make' directly, all command-line options are forwarded to the build tool

Requirements

  • rmake is written in Bash, so a Bash interpreter is required.
  • rmake requires the getopt tool.
  • It makes remote connections via ssh and replicates using rsync, so both mush be installed and the master host and the build servers.
  • An up-to-date SVN client and xsltproc are recommended for the SVN pedantic exclusion filter.
  • Non-interactive SSH logins to the build servers is also required.
  • If using a windows machine as your development machine then you should have cygwin installed before beginning the install instructions.
  • Also, set up a build server before starting the install.
  • The time on the build server must not be ahead of the time on your workstation, nor is it allowed to be more than 5 minutes behind. See the section below on Time.

Definitions

Workstation: The computer that you normally use for editing your source files. Not necessarily the same as the Build Server. Can be any OS (Linux, BSD, Windows, etc).

Build Server: This is the computer where your build is actually done. This Build Server must be of the approved Linux Kernel release with the necessary patches, etc for the PolyServe product. The Build Server isn't anything special. It's just a machine that is most commonly prepped by easy-rmake. rmake copies files from the SVN tree and runs the build on the Build Server. So, basically, a Build Server is any computer running a sanctioned, approved, blessed version of Linux. You need to have a local, non-root account on it (but easy-rmake.sh will make one for you if you can provide credentials for easy-rmake to login to the build server as root).

easy-rmake: The automated rmake installation script. See the Guided Install section.

Platform: In rmake a "platform" is a name used to identify a build configuration on the command-line and in the configuration files. An example of a platform name is "rhel5". These are pre-defined in .rmake file. You can find .rmake under the project root dir. Ex: trunk/sns/.rmake

For more definitions please see https://swdnaseng.usa.hp.com/plan/technology/Build/BuildFAQs.html.

Scenario 1: Development Workstation + Build Server

Your workstation is a Linux or Windows box. You happen to have a RHEL5 Linux box sitting in the next room that happens to have the correct Linux kernel or your site provides access to a nightly build server like rhel5-2-64.pdx.polyserve.com. You may or may not have exclusive access to it. You edit files from your workstation. You run easy-rmake to set up your build environment for rmake. When it comes time to build something, you run rmake which performs the build on your Build Server, the Linux box in the next room.

Scenario 2: Solo Build Server

You have a Linux machine that is going to also be development Workstation and your Build Server. You have read the Advanced Configuration section make sure that the build happens outside of your local SVN tree. If you haven't, rmake will notice that and abort.

CAUTION: Developing on a Build Server is not recommended! You should store your source code under development in a regularly backed-up location. In Beaverton it is recommended to develop on comp1/comp2 (psssh1.cce.hp.com) and/or CIFS/NFS mount your svn tree hosted on the ds cluster.

Guided Install with easy-rmake.sh

The Guided Install is intended for new developers or those that would like to use the automated install script to assist the setup of a new build machine.

To run the Guided Install on your development machine/workstation simply type or paste the following command:

bash -f <(svn cat https://swd-nas.csd200a.com/svn/swd-nas/trunk/tools/rmake/HP/easy-rmake.sh )
Note 1: During the script when asked for a build server, please enter your user information as well as build server name using this format: user-name@buildserver. If you don't have a user on the build server, then either create a user or use root. Building as root is not recommended.
Note 2: In Cygwin there might be a problem with svn interacting with the user, in that case try:

bash -c "$(svn cat https://swd-nas.csd200a.com/svn/swd-nas/trunk/tools/rmake/HP/easy-rmake.sh )"|

Caveat: On RHEL5.x, if you come across a problem with the "Certificate information", execute just the part inside the parenthesis first. Then, re-execute the entire "bash -f (....." command.
Note 3: If your build server is also your workstation, you will need to read the Advanced Configuration section below,

Manual Install

Most developers that aren't yet using rmake have one or more SVN workspaces with which they do development and SVN transactions with. There are ways to copy one of these workspaces to a new location on a machine that will act as the master server, but a previously polluted workspace is not a good starting point. https://swd-nas.csd200a.com/svn/swd-nas/trunk/tools/svntools/svntools provides some tools to purify an SVN workspace, but it is recommended that a new workspace be checked out on from scratch.

Note: svntools is checked out and sourced by your .bashrc automatically by easy-rmake.

Manual Step 1: Checkout

It is recommended that a new SVN workspace be created based on the following best practice:

Change to your home directory as a starting point and then recursively build a replica of the subversion repository like so:

svn checkout -N https://swd-nas.csd200a.com/svn/swd-nas svn

svn update -N svn/trunk svn/branches svn/branches/private svn/branches/devel svn/trunk/tools
svn update svn/trunk/tools/rmake svn/trunk/tools/svntools

Manual Step 2: Project Checkout

From this point you can proceed to checkout your required projects, for example:

cd svn/trunk

svn update sns

Manual Step 3: Add to Path

Either:

  • Add ~/svn/trunk/tools/rmake to your path
    • (Optional) Add the following to your ~/.bashrc |echo "alias rmake='rmake-color'" >> ~/.bashrc|
  • Create a symlink /bin/rmake -> ~/svn/trunk/tools/rmake/rmake-color and add ~/bin to your path. | mkdir ~/bin && ln -s ~/svn/trunk/tools/rmake/rmake-color ~/bin/rmake && echo 'PATH=$PATH:/bin' >> ~/.bashrc'|

Manual Step 4: Setup Bash Completion (Optional)

Create a link /etc/bash_completion.d/rmake that points to rmake/bash_completion.d/rmake

sudo ln -s ~/svn/trunk/tools/rmake/bash_completion.d/rmake /etc/bash_completion.d/rmake

Manual Step 5: Create .rmakerc

Continue reading through the next section to learn about the .rmakerc file.

The .rmakerc File

The .rmakerc file is usually found in your home directory at ~/.rmakerc. This file is really just a common place to store the configuration of your build server platforms.

The typical ~/.rmakerc file defines the following components for a platform:

platform=user@server
  • "platform": the platform name
  • "user@": the user to ssh to the Build Server as (optional)
  • "server": the Build Server to synchronize and build with

For example:

rhel5=ewoodruff@tmr11s2rbvm

Advanced Configuration

In reality, rmake will find these platform identifiers from the shell environment so the .rmakerc file is not essential.

Additionally, a .rmakerc need not be placed in your home directory, but rather it can be placed in a project root directory as well. If placed in a project root such as trunk/sns/.rmakerc, any configured settings defined in ~/.rmakerc will be inherited but can also be overridden in the project-specific version.

Typically a project-specific .rmakerc file is used to specialize the configuration for that project, such as defining a specific build path on the Build Server. See the following definitions of the allowed forms of platform definitions in rmake:

Form 1: Remote Host

platform=user@server:build/path
  • "platform": the platform name
  • "user@": the user to ssh to the Build Server as (optional)
  • "server": the Build Server to synchronize and build with
  • "build/path": the location on the Build Server to place the synchronized files and perform a build (optional)
Note: Typically the build/path should not be specified. When left unspecified, rmake will automatically select a path on the Build Server that is a moral equivalent to the path to the project root on the local workstation. This allows the rmake user to not have to specify a new .rmakerc configuration for each private branch that is being developed.
Tip: If you are using rmake directly on a Build Server (Scenario 2 above, not recommended), then you will need to specify a build path because otherwise rmake will automatically choose the same path as the svn tree, which is not allowed. In that case, you can just use Form 2 below.

Form 2: Local Host

platform=~/build/path
  • "build/path": the location on the local workstation to place the synchronized files and perform a build (optional)

Lastly, the RMAKE_PROXY option can be specified if there is another location that hosts the svn tree (besides your workstation) that would be a more efficient place to rsync from. For example, my svn tree is actually NFS mounted to the ds cluster, so I proxy through comp1:

RMAKE_PROXY=ewoodruff@comp1
Tip: The same rules apply to RMAKE_PROXY as a platform identifier, meaning this option supports Form 1 and Form 2 and can be overridden in a project-specific .rmakerc.

Usage Guide

rmake Help

rmake is a high-level make program that replicates a master workspace

to a remote build server and runs make.

Usage: rmake -a [OPTION]... [TARGET]...
rmake -p "PLATFORMS" [OPTION]... [TARGET]...

Options
-a operate on all platforms in RMAKE_PLATFORMS
-p, --platform=NAMES operate on NAMES
-l list configured platforms.
-r, --resource print the configured resource for a platform
-k, --keep-going continue on platform error
-P, --parallel parallelize (implies -qk)
-u, --update-only update remote workspaces, but do not make
-d, --delete delete extraneous files from destination dirs
-D, --delete-only delete but transfer no files (implies -du)
-i, --pedantic ignore/exclude files not in good standing with svn
-m, --svn-meta include svn metadata (.svn/ directories)
-F, --file-list=FILES synchronize FILES instead of RMAKE_FILE_LIST
-R, --relative synchronize current dir instead of RMAKE_FILE_LIST
-C, --directory=DIR change to DIR before synchronizing or making
-L, --log-dir=DIR write PLATFORM.log files to DIR/WORKSPACE_ID
-A, --append append to logs rather than rotating them
-x, --exec=COMMAND run "COMMAND platform exitcode timestamp server log"
-w, --where-am-i print the located workspace root
--no-decorate don't decorate/annotate build output
-c only run rmake-check
-q suppress non-error messages
-v increase verbosity (ex. use rsync -v)
--version print SVN revision
-h, --help show this help|

Use Case 1: Perform a Build in a specific directory

cd trunk/klondike/src/common

rmake -p rhel5 clean all

Use Case 2: Perform the same build on all configured platforms (RMAKE_PLATFORMS)

rmake -a clean all

Use Case 3: Purge all build-derived files from a specific directory and rebuild

cd ../mxauthz

rmake -Rdi all

Caution: Make sure you know what you are doing; using this command by mistake could require a full rebuild.

Use Case 4: Use rmake with vim

:set makeprg=TERM=dumb\ rmake\ --no-decorate\ -prhel5

:make

Use Case 5: Use rmake without an .rmakerc

rhel5=user@host:path/to/build rmake -p rhel5 rpm

Time

When building on a different server than you are developing, time can be an issue, especially for incremental builds. It is very important to remove time differences between your Workstation and your Build Server. NTP can be useful, but nevertheless, virtual machines can sometimes have problems keeping time anyway. rmake will check for a time differential in the following ways:

  1. The Build Server may not be in the future, otherwise builds might always be up-to-date.
  2. The Build Server may not be more than 5 minutes is the past. The further is the past the Build Server is, the more it will unnecessarily rebuild.
Note: When performing a full rebuild from scratch with '-d', rmake has no need to check the time differential unless '-R' was used.
Tip: Here is an easy to set the time on your Build Server (if you have root access):

date | ssh root@server xargs -0 date -s

A Bash alias can also be created to do this frequently:
alias rmake='date | ssh root@server xargs -0 date -s; rmake'|
